# Neo-MoFox Loop 重构落地方案 v2

> **版本**: v2.0  
> **日期**: 2026-03-26  
> **性质**: 可执行的工程落地方案  
> **目标**: 在不预定义具体状态/任务态的前提下，先搭出一个可被插件自由扩展与统一治理的骨架系统

---

## 0. 本版修正的核心方向

v1 的问题在于把很多未来能力提前写死了：任务态、状态变量、Agent 类型、调度语义，都被核心层预先定义。  
这不符合 Neo-MoFox Loop 的真实路线。

v2 重新定义为：

- **核心层只负责骨架，不负责语义**
- **状态、任务态、Agent、上下文切片，都由插件提供**
- **调度器只认识注册表和能力概览，不预设未来会出现什么**
- **系统启动时可以很空，靠插件逐步长出能力**

换句话说，Neo-MoFox Loop 不是“先设计好所有状态再实现”，而是“先实现一个能容纳状态生长的骨架”。

---

## 1. 设计总则

### 1.1 骨架优先，不预设业务

核心层只提供以下通用能力：

- 身份核
- 状态总线
- 调度中枢
- 插件注册与生命周期管理
- 上下文转移与注入裁剪
- 事件与日志观测
- 持久化与恢复

核心层不内建：

- 固定任务态列表
- 固定状态变量列表
- 固定 Agent 类型列表
- 固定的“聊天/深思/内作”语义实现

这些都由插件决定。

### 1.2 插件化优先

系统中的大多数“能力”都应该以插件形式出现，包括：

- 状态插件
- 任务态插件
- Agent 插件
- 上下文切片插件
- Prompt 注入插件
- 调度策略插件
- 表现层插件

### 1.3 统一治理优先

插件可以自由扩展，但必须进入统一治理：

- 统一注册
- 统一可见
- 统一生命周期
- 统一状态真相
- 统一注入门控
- 统一调度入口

同时，所有 Agent 必须是**独立上下文、独立工具集**的执行单元：

- 每个 Agent 维护自己的 messages / short-term memory
- 每个 Agent 只暴露自己的工具白名单
- 调度器只能通过注册信息选择 Agent，不能共享一套全局 Agent 上下文

---

## 2. 目标架构

```
外界输入
  ↓
统一入口层
  ↓
统一调度中枢
  ↓
插件注册表 / 能力概览
  ↓
任务态插件 / 状态插件 / Agent 插件 / Prompt 插件
  ↓
上下文转移与最小注入
  ↓
主模型输出
```

在这个架构里：

- 核心层不关心“现在一定有什么任务态”
- 调度器只关心“当前注册了什么、谁可用、谁适合处理”
- 插件决定“我是谁、我提供什么状态、我能处理什么任务”

---

## 3. 与 v1 的关键差异

### 3.1 不再预设任务态

v1 里把聊天态、深思态、内作态、检索态等画成了固定模块。  
v2 不这么做。

v2 中，任务态不是核心层的枚举，而是插件注册出来的行为单元。  
系统一开始可以没有任何业务任务态，后续通过插件逐步添加。

### 3.2 不再预设状态类型

v1 里把情绪、意图、关系、任务负载等写成了固定状态分层。  
v2 中，状态不是核心层内建分类，而是插件向统一状态系统声明自己的状态切片。

### 3.3 不再预设 Agent 类型

v1 里把记忆 Agent、资料 Agent、工具 Agent 写成了固定构件。  
v2 中，Agent 也是插件产物，调度器只看它的能力标签、依赖、可用性和上下文边界。

---

## 4. 核心模块边界

### 4.1 Identity Core

职责只保留：

- 维护统一身份核
- 保护“我还是我”的连续性
- 提供身份一致性校验

不负责：

- 具体人格内容
- 具体状态变量
- 具体任务态实现

### 4.2 Plugin Runtime

这是 v2 的中心。  
插件运行时负责：

- 发现插件
- 加载插件
- 注册插件能力
- 管理插件生命周期
- 汇总插件可见状态
- 汇总插件可用任务态
- 汇总插件可调用 Agent

插件运行时还必须维护每个 Agent 的隔离边界：

- 独立上下文
- 独立工具集
- 独立短期记忆
- 独立生命周期

调度器可以看见概览，但不能直接把多个 Agent 的上下文拼成一个共享脑。

### 4.3 Unified State Bus

状态总线不是一堆固定状态类，而是一个统一通道：

- 插件可以声明状态片段
- 插件可以更新状态
- 插件可以请求状态切片
- 调度器可以查看状态概览

### 4.4 Unified Scheduler

调度器不预设任务态名字。  
它只做四件事：

- 看当前有哪些插件和能力
- 判断是否需要切换任务态
- 判断是否需要询问某个 Agent
- 决定结果回流给主意识的方式

调度器只做编排，不共享 Agent 的执行上下文。  
它通过能力标签选择谁来做事，通过回流接口接收结果。

### 4.5 Prompt Assembler

Prompt 组装器是统一注入出口。  
它负责把：

- 身份核
- 状态切片
- 任务态信息
- Agent 回流
- 记忆摘要
- 当前目标

按预算组装成主模型上下文。

## 5. 插件化定义

### 5.1 插件是能力单元

插件不只是“功能模块”，而是系统能力的最小承载单元。  
一个插件可以提供：

- 状态
- 任务态
- Agent
- Prompt 片段
- 调度策略
- 事件监听器
- 工具接口

其中，Agent 插件必须声明自己的：

- 独立上下文策略
- 独立工具集
- 独立短期记忆策略
- 独立结果回流格式

### 5.2 插件必须可声明

建议为每个插件提供清晰的清单式元信息，例如：

```yaml
id: memory_plugin
category: state_plugin
provides:
  - state.memory.summary
  - state.memory.slice
hooks:
  - on_boot
  - on_tick
  - on_state_request
dependencies:
  - identity_core
priority: 50
```

核心层不需要知道插件内部怎么实现，只需要知道它“提供了什么”和“如何参与治理”。

### 5.3 插件分类

推荐的插件类别不是固定业务能力，而是治理视角：

- `state_plugin`：声明和维护状态切片
- `task_plugin`：声明一个可调度任务态
- `agent_plugin`：提供一个可独立执行的 Agent
- `prompt_plugin`：贡献 prompt 段或注入规则
- `scheduler_plugin`：提供调度策略或触发逻辑
- `tool_plugin`：提供工具能力
- `presentation_plugin`：提供视觉/语音/表现层能力

对 Agent 类插件，还应额外声明：

- `context_scope`：上下文范围
- `tool_scope`：工具白名单
- `memory_scope`：短期记忆范围
- `output_scope`：回流输出格式

这些类别只是治理标签，不是核心预置业务。

## 6. 状态、任务态、Agent 的生成方式

### 6.1 状态不是写死的

状态不是核心层的固定枚举。  
它应由插件声明、注册、更新和持久化。

例如：

- 某个插件可以声明“关系状态”
- 某个插件可以声明“情绪状态”
- 某个插件可以声明“任务压力状态”
- 某个插件可以声明“时间感状态”

核心层只负责统一接收和统一裁剪。

### 6.2 任务态不是写死的

系统一开始不必有“聊天态、深思态、内作态”的硬编码。  
它们可以在运行时作为插件出现。

调度器的职责不是识别某个预设任务态，而是：

- 看插件能力标签
- 看当前上下文缺口
- 看当前任务是否需要切换
- 选择合适的任务态插件

### 6.3 Agent 不是写死的

记忆、资料、工具、文件等都可以是 Agent 插件。  
调度器不直接依赖某个具体 Agent 名称，而依赖其能力注册。

每个 Agent 默认都拥有：

- 自己的上下文
- 自己的工具集
- 自己的短期记忆

这三者彼此隔离，只有结果回流进入统一调度与统一状态系统。

## 7. 运行流程

### 7.1 启动阶段

1. 装载最小骨架。
2. 扫描插件目录。
3. 构建插件注册表。
4. 汇总可见状态、任务态、Agent、Prompt 片段。
5. 完成统一调度与注入入口初始化。

### 7.2 运行阶段

1. 外界输入进入统一入口。
2. 调度器读取注册表和状态概览。
3. 调度器决定当前是否需要某个插件能力。
4. 插件产出状态、候选信号或执行结果。
5. 调度器裁剪后把必要信息送入主模型。

如果调用的是 Agent 插件，则该 Agent 在自己的隔离上下文中运行，并且只能使用自己的工具白名单。

### 7.3 回流阶段

1. Agent 或任务态完成工作。
2. 结果先进入统一回流层。
3. 调度器决定是否合并进状态系统。
4. 最终再决定是否暴露给主意识。

回流时只传结构化结果，不回传整个 Agent 执行上下文。

## 8. 重构路线图

### 阶段 1：搭骨架

目标是先把“能装插件、能看插件、能管插件”做好。

- 插件注册表
- 插件生命周期
- 状态总线
- 调度中枢
- Prompt 组装器
- 观测与日志

### 阶段 2：让插件接管语义

把原来写死在核心层的语义逐步移出核心：

- 状态类型移入插件
- 任务态移入插件
- Agent 移入插件
- Prompt 段移入插件

### 阶段 3：统一治理

做跨插件统一治理：

- 状态冲突解决
- 任务态切换门控
- 回流裁剪
- 优先级排序
- 可观测审计

### 阶段 4：渐进替换旧实现

把现有功能逐步迁移到插件层，不一次性重写所有东西。

## 10. 风险与缓解

### 10.1 风险

- 插件过多后治理失控
- 状态定义冲突
- 任务态命名不统一
- 调度器背负太多语义判断
- 主意识被过量插件信息淹没

### 10.2 缓解

- 严格插件元信息
- 统一状态真相
- 统一回流裁剪
- 统一优先级与生命周期
- 主意识只看最小切片

---

## 11. 验收标准

### 11.1 架构验收

- 核心层没有预设固定任务态
- 状态可以由插件自由注册
- Agent 可以由插件自由注册
- 调度器能看见全局概览
- 主意识只接收裁剪后的必要信息

### 11.2 哲学验收

- 统一身份核没有被多中心化
- 状态变化可以持续递归
- 潜意识可以并行，但不会越权
- 系统能表现出跨状态连续生活感

---

## 12. 结论

v2 的关键变化只有一句话：

**Neo-MoFox Loop 不是先定义好所有状态和任务态再实现，而是先实现一个能让状态、任务态、Agent 以插件方式生长出来的骨架。**

因此，重构的重点不是“把功能写完”，而是：

- 把插件化治理做成第一公民
- 把状态/任务态/Agent 的语义从核心层移出去
- 把统一调度和统一注入做成唯一主干

只有这样，系统后续长出来的任何状态和任务态，才不会把骨架本身撑裂。
