# EchoBot Live2D 面板借鉴纪要

> 说明：本文记录可借鉴的面板能力与架构启发，不涉及当前阶段的具体实现细节。

## 1. 总体结论

EchoBot 的 Live2D 能力在“前端表现层”完成度较高，  
尤其是舞台视觉、模型交互、口型联动这三块，具有直接借鉴价值。

但它当前更像“高质量可视化面板”，还不是“可被智能体编排的角色执行层”。

## 2. 可借鉴的核心能力

### 2.1 舞台渲染层是分层结构

- 角色层、背景层、粒子层、后处理效果层分离。
- 光影、暗角、颗粒、色调等参数有统一状态管理。
- 价值：后续做情绪状态映射时，可把“内在状态”映射到“舞台风格”，而不是只改文本语气。

参考：

- `/root/Elysia/EchoBot/echobot/app/web/modules/live2d.js`
- `/root/Elysia/EchoBot/echobot/app/web/index.html`

### 2.2 模型交互体验完整

- 模型加载、拖拽、缩放、视线跟随、重置视角都具备。
- 模型切换与上传有配套 UI 与状态回退逻辑。
- 价值：可直接作为“身体层交互基线”，减少重复造轮子。

参考：

- `/root/Elysia/EchoBot/echobot/app/web/modules/live2d.js`

### 2.3 口型同步链路成熟

- 使用 TTS 音频分析（RMS）驱动嘴部参数，平滑更新。
- 不是简单开关式嘴型，而是连续值驱动。
- 价值：语音存在感明显提升，可作为后续“主动说话”形态的基础。

参考：

- `/root/Elysia/EchoBot/echobot/app/web/modules/tts.js`
- `/root/Elysia/EchoBot/echobot/app/web/modules/live2d.js`

### 2.4 资源管理与安全边界清晰

- 后端负责模型发现、上传校验、路径安全、资源路由。
- 前端负责渲染与交互，职责边界明确。
- 价值：更适合长期迭代，不易因为视觉层升级污染主业务层。

参考：

- `/root/Elysia/EchoBot/echobot/app/routers/web.py`
- `/root/Elysia/EchoBot/echobot/app/services/web_console.py`

## 3. 当前借鉴边界

- EchoBot 的 Agent 侧没有原生 Live2D 控制工具体系。
- Live2D 主要受前端本地交互与音频链路驱动，不是由智能体任务流统一编排。
- 因此可借“面板能力”，不能直接等同于“智能体可控身体系统”。

## 4. 对 Neo-MoFox Loop 的启发

### 4.1 可先借“身体层”，再接“意识层”

- 先复用其舞台和模型交互思想，形成稳定可视化身体层。
- 再让主意识/内作层通过统一接口驱动身体表现。

### 4.2 建议采用“编排中台”而非直连

- 方向建议：`主意识/Agent -> 动作编排层 -> 前端执行层`。
- 编排层负责白名单、节流、冲突仲裁、优先级。
- 这样能避免多个状态同时抢控 Live2D 导致行为抖动。

### 4.3 MCP 可以作为能力入口，但不应直达前端细节

- MCP 适合承载“标准化动作命令”。
- 前端细节（参数名、模型差异）应被编排层封装，避免主意识直接依赖低层参数。

## 5. 一句话总结

EchoBot 的 Live2D 面板值得借鉴的是“高完成度身体表现层”；  
下一步关键不是复制界面，而是把它升级为“可被主意识有序驱动的执行层”。
