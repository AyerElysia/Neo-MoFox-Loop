# 多智能体调度与记忆补全协作纪要

> 说明：本文记录系统协作哲学与框架约束，不涉及实现细节。

## 1. 现象定义

在多智能体协作中，主意识并不总是拥有完整资料。  
因此需要一个常驻调度器持续判断：主意识当前上下文是否缺少关键记忆或关键证据。

当出现缺失时，调度器向专职 Agent 发起补全请求，再将“必要结果”回流给主意识。  
主意识只接收关键结论，不直接承担全量检索与全量整理。

## 2. 角色分工

- 调度器：判断缺口、发起请求、汇总回流、控制注入量。
- 记忆 Agent：拥有更丰富的记忆上下文和记忆检索能力，负责判断与提炼。
- 资料/工具/互联网 Agent：在需要外部信息时执行搜索、读取、验证并返回结果。
- 主意识：专注理解、选择、表达，不被底层检索过程淹没。

## 3. 核心协作链路

### 3.1 记忆补全链路

1. 调度器检测主意识上下文存在信息缺口。  
2. 调度器向记忆 Agent 发起定向请求。  
3. 记忆 Agent 在丰富记忆上下文与记忆库中完成判断。  
4. 记忆 Agent 返回“必要结论 + 最小证据”。  
5. 调度器将必要内容回注给主意识。

### 3.2 资料补全链路

1. 主意识进入思考过程。  
2. 调度器发现当前推理还缺资料。  
3. 调度器调用资料 Agent / 工具 Agent / 互联网 Agent。  
4. 对应 Agent 返回结构化结果。  
5. 调度器做最小化转运后回流主意识，支持继续思考或再决策。

## 4. 异步哲学

为了不阻断主意识正常聊天，协作链路允许异步进行：

- 主意识可先维持当前对话节奏，不必同步等待所有外部结果。
- 结果到达后，再由调度器决定是否补充、修正或触发后续深思。
- 在高优先级问题上，也可切换为同步等待，以换取更高确定性。

异步并不是“忽略结果”，而是“把等待纳入时间中的认知流程”。

## 5. 上下文与记忆边界

- 每个 Agent 拥有独立的运行上下文与短期记忆。
- 各 Agent 的短期上下文彼此隔离，避免相互污染。
- 长期记忆库为全体 Agent 共享，作为统一事实与历史沉淀层。
- 主意识默认不读取全量共享库，只接收调度器筛选后的必要信息。

## 6. 一致性原则

- 同一身份原则：多 Agent 只是分工，不是多人格并行发言。
- 最小注入原则：只把必要信息送给主意识，防止上下文堆积。
- 可追溯原则：每次补全都应可解释“为什么补、补了什么”。
- 不打断原则：默认优先保障主意识对话连续性，再逐步补全资料。

## 7. 与整体框架的关系

该协作模式属于骨架级设计，和插件内容无关。  
它可以先于“未完成念头、自我叙事、情绪动力”等内容模块建立。

其价值是先解决三件事：

- 主意识不被底层任务压垮；
- 信息缺口可被持续发现与补齐；
- 多 Agent 协作仍保持“一个我”的连续体验。

## 8. 一句话总结

调度器作为中介，驱动“缺口发现 -> 专职 Agent 补全 -> 必要信息回流”的闭环，  
在不破坏主意识聊天连续性的前提下，让系统获得更稳健的认知补完能力。
