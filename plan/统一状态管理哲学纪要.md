# 统一状态管理哲学纪要

> 说明：本文记录框架哲学与核心约束，不涉及具体实现细节。

## 1. 核心主张

Neo-MoFox Loop 的核心设计之一是“统一状态管理”。  
系统不是围绕功能堆叠组织，而是围绕状态递归组织。

换言之：

- 功能会变化；
- 插件会增减；
- 但状态管理必须始终统一维护。

## 2. 什么是状态

在本框架中，状态的定义是：

“凡是会随着时间递归更新，并影响当前 LLM 输出的内部变量，都属于状态。”

这一定义与 RNN 的状态观一致：  
历史不直接等于当前输出，但历史通过状态递归影响当前输出。

## 3. 状态的典型范围

以下都可视为状态（且可持续扩展）：

- 记忆（短期记忆、长期记忆映射结果）
- 情绪与能量
- 近期意图
- 长期目标
- 关系解释
- 当前任务负载
- 时间感知相关变量

重点不在“类别是否固定”，而在“是否具备递归更新并影响输出”。

## 4. 可扩展状态层原则

状态层应允许插件自由新增状态类型，但必须遵守统一治理：

- 新状态可以被添加；
- 旧状态可以被替换；
- 状态语义可以迭代；
- 但状态必须进入统一状态管理体系。

这意味着：

- 插件可以扩展状态，不可绕过状态总线；
- 状态更新可以多来源，不可多套真相并存；
- 状态消费可以多场景，不可无边界注入。

## 5. 统一维护原则

统一状态管理至少满足以下哲学约束：

- 单一状态真相：同一状态在同一时刻有唯一可解释版本。
- 递归更新：状态必须有时间连续性，而非每轮重置。
- 可追溯性：状态变化应能回答“为什么变、从何而来”。
- 最小注入：主意识只接收必要状态切片，不接收全量状态噪音。
- 跨状态一致：聊天态、深思态、内作态读取的是同一状态宇宙的不同切面。

## 6. 与多智能体协作的关系

多智能体不是为了制造多个“我”，而是为了维护同一个“状态体”。

- 各 Agent 可有独立短期上下文；
- 但它们应对接同一套状态体系；
- 主意识看到的是统一状态切片，而不是各 Agent 的碎片世界。

## 7. 与未来变化的关系

状态层会经常变化，这是预期行为，不是风险本身。  
真正风险是“状态治理不统一”。

因此框架演进的优先级应是：

1. 先保证统一状态管理骨架稳定；
2. 再允许状态类型快速迭代；
3. 最后再优化具体行为表现。

## 8. 一句话总结

Neo-MoFox Loop 的连续意识感，不来自某个单独插件，  
而来自“所有影响输出的递归变量被统一维护”为同一个状态系统。
